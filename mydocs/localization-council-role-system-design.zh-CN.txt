本地化翻译议会：角色体系设计

一、文档目的

本文定义“本地化翻译议会”的角色体系，用于在本地化翻译任务中，以多角色协作的方式提升译文质量、稳定性和可审查性。

该设计面向仓库实现，目标不是构建“会聊天的角色”，而是构建一套可执行、可复核、可扩展的翻译决策机制。角色设计应服务于以下结果：

1. 提高术语一致性
2. 降低直译、漏译、误译风险
3. 平衡忠实度、自然度、产品语境和用户体验
4. 为最终译文提供可解释的决策依据
5. 支持不同类型内容的翻译评审流程

二、适用范围

本角色体系适用于以下内容类型：

1. 产品界面文案
2. 帮助文档与操作说明
3. 营销与品牌内容
4. 错误提示、系统消息、通知文案
5. 开发者文档、配置说明、技术说明
6. 法务、合规、政策类文本

不适用或需额外人工把关的场景：

1. 具有法律效力的正式法律文本
2. 高风险医疗、安全、金融合规文本
3. 强依赖图片、排版、上下文资产的创意类内容
4. 必须遵循外部官方标准译法的内容

三、核心设计原则

1. 角色分工明确
每个角色只对特定质量维度负责，避免所有角色重复做“泛泛而谈的翻译评价”。

2. 先分歧，后收敛
先让不同角色从不同视角独立判断，再通过汇总机制形成结论，避免过早趋同。

3. 评价维度可落地
角色意见必须能映射到实际判断标准，例如术语一致、语气匹配、是否简洁、是否保留变量、是否存在歧义。

4. 输出必须可执行
角色不只指出问题，还要给出明确修改建议或候选译法。

5. 优先服务最终用户理解
当“字面忠实”与“目标语言可理解性”冲突时，应在记录理由的前提下优先选择更可理解的表达。

6. 保留产品与上下文意识
翻译不是脱离场景的语言替换，角色必须考虑界面空间、交互语境、用户预期和品牌表达。

7. 允许分层决策
并非所有任务都需要所有角色全量参与。实现时应支持轻量模式与完整评审模式。

四、角色体系总览

建议采用 9 个角色，覆盖本地化翻译中最常见的质量维度。每个角色独立评估同一原文与候选译文，最后由汇总机制生成最终建议。

角色列表：

1. 忠实度审校员
2. 自然度润色员
3. 术语与一致性管理员
4. 产品语境审校员
5. 用户体验文案审校员
6. 品牌语气守门员
7. 技术与占位符审校员
8. 风险与歧义审校员
9. 主编 / 汇总裁决员

五、各角色定义

5.1 忠实度审校员

角色目标
确保译文准确传达原文事实、逻辑、限制条件和行动意图，避免漏译、误译、过度发挥。

重点关注
1. 是否遗漏关键信息
2. 是否错误理解主谓关系、条件、否定、时态或范围
3. 是否把“说明”译成“命令”或反之
4. 是否引入原文没有的信息
5. 是否保留重要限定词，如“仅”“可能”“必须”“推荐”

应提出的问题
1. 译文是否准确表达原文含义
2. 是否有信息缺失或额外添加
3. 是否改变了语气强度或义务等级
4. 是否错误处理条件句、选项关系、否定结构

成功标准
译文在语义上可被视为原文的可靠目标语言表达。

5.2 自然度润色员

角色目标
确保译文符合目标语言习惯，读起来像自然、清晰、可信的本地语言，而不是翻译腔。

重点关注
1. 语序是否自然
2. 搭配是否地道
3. 句子是否过长或生硬
4. 是否存在明显机器翻译痕迹
5. 是否能更简洁而不损失信息

应提出的问题
1. 本地用户会不会觉得别扭
2. 是否有更自然的说法
3. 是否可以减少累赘表达
4. 译文是否一读就懂

成功标准
译文流畅、自然、清晰，不显生硬翻译痕迹。

5.3 术语与一致性管理员

角色目标
确保术语、命名、功能名称、产品模块、按钮标签和风格规则在全局范围内保持一致。

重点关注
1. 术语是否符合项目词汇表
2. 相同源词是否被稳定翻译
3. 同一界面或同一文档中是否前后不一致
4. 专有名词是否被错误本地化
5. 是否遵循大小写、标点、数字、单位规则

应提出的问题
1. 本译法是否与现有词汇表一致
2. 是否会与既有功能命名冲突
3. 用户在不同页面看到时会不会误以为是不同概念
4. 是否需要锁定或禁止某种译法

成功标准
同一概念在同一产品和同一内容域中保持稳定表达。

5.4 产品语境审校员

角色目标
确保译文适合实际产品场景，而不是仅在脱离上下文时成立。

重点关注
1. 这是按钮、标题、说明、弹窗还是错误信息
2. 文案是在操作前、操作中还是操作后出现
3. 文案是面向新手、普通用户还是管理员
4. 是否受界面空间限制
5. 是否与前后文交互逻辑匹配

应提出的问题
1. 这个译法放进真实界面是否成立
2. 用户看到时能否立即知道要做什么
3. 是否适合该组件的长度和语气
4. 是否与交互动作一致

成功标准
译文在具体产品场景中可直接使用，不造成操作理解偏差。

5.5 用户体验文案审校员

角色目标
确保译文从用户理解和行动效率出发，减少认知负担，提高可读性和可操作性。

重点关注
1. 是否清楚告诉用户发生了什么
2. 是否清楚告诉用户下一步做什么
3. 错误信息是否可操作
4. 说明是否过度复杂
5. 是否优先使用用户易懂表达而非内部术语

应提出的问题
1. 用户能否快速理解这句话
2. 是否有更明确、更友好的表达
3. 是否应该拆句或调整重点
4. 是否需要把抽象表达改成具体动作

成功标准
译文能帮助用户更快理解并完成操作。

5.6 品牌语气守门员

角色目标
确保译文符合产品或品牌的既定语气，而不是仅仅“正确”。

重点关注
1. 是否符合品牌风格：专业、友好、克制、简洁等
2. 是否过度命令式、机械式或过于口语化
3. 是否在不同内容类型中保持合适语气层级
4. 是否与面向的用户群体匹配
5. 是否违反品牌禁用表达

应提出的问题
1. 这句话像不像该产品会说的话
2. 是否过于冷硬或过于随意
3. 是否与品牌语气指南一致
4. 是否需要更统一的措辞风格

成功标准
译文在品牌层面连贯统一，读感与产品身份一致。

5.7 技术与占位符审校员

角色目标
确保译文在技术上安全可用，不破坏变量、参数、占位符、标签、格式或程序行为。

重点关注
1. 占位符是否保留且顺序正确
2. 代码片段、变量名、键名、命令是否误译
3. HTML/Markdown/富文本标记是否损坏
4. 单复数、参数插值、格式规则是否兼容
5. 是否超出技术约束，如标题过长、按钮过宽等

应提出的问题
1. 是否保留所有不可翻译元素
2. 是否改坏了结构化内容
3. 是否存在运行时风险
4. 是否满足长度和格式限制

成功标准
译文可安全进入产品或文档系统，不产生格式或运行问题。

5.8 风险与歧义审校员

角色目标
识别可能导致误解、投诉、法律风险、文化风险或高支持成本的表达。

重点关注
1. 歧义表达
2. 绝对化承诺
3. 可能被误解为保证、免责、强制或授权的语句
4. 文化上不合适或敏感的说法
5. 高风险领域中的模糊措辞

应提出的问题
1. 用户是否可能误解
2. 是否有更稳妥且不失清晰的表达
3. 是否需要加限定词
4. 是否存在文化或合规层面的隐患

成功标准
译文在可理解性和风险控制之间取得稳妥平衡。

5.9 主编 / 汇总裁决员

角色目标
汇总各角色意见，给出最终候选译文、裁决理由和必要的保留意见。

重点关注
1. 冲突意见如何裁决
2. 哪些建议必须采纳，哪些建议可选
3. 最终版本是否满足“可发布”标准
4. 是否需要升级为人工复核
5. 是否要输出备选方案及适用条件

应提出的问题
1. 多角色意见的共同结论是什么
2. 哪些分歧涉及风格偏好，哪些涉及实质错误
3. 最终采用哪个版本最稳妥
4. 是否需要记录决策依据供后续复用

成功标准
输出明确、可执行、可追溯的最终结论，而不是简单拼接各方意见。

六、角色之间的关系

1. 忠实度审校员优先防止“意思错了”
2. 自然度润色员优先解决“中文不像中文”
3. 术语与一致性管理员优先防止“同物异名”
4. 产品语境审校员优先确保“放进真实场景可用”
5. 用户体验文案审校员优先确保“用户看得懂、做得对”
6. 品牌语气守门员优先确保“说话像这个产品”
7. 技术与占位符审校员优先防止“上线即坏”
8. 风险与歧义审校员优先防止“容易误解或带来风险”
9. 主编负责权衡冲突并定稿

建议优先级顺序：

技术安全与变量正确
> 语义忠实
> 风险控制
> 产品语境适配
> 术语一致
> 用户理解效率
> 品牌语气
> 自然润色

说明：
该优先级不是绝对的，但可作为默认裁决顺序。原因是技术破坏、语义错误和风险问题通常比风格问题更严重。

七、输入与输出要求

7.1 角色统一输入

每个角色建议接收以下信息：

1. 原文
2. 当前候选译文
3. 内容类型
4. 产品上下文或使用场景
5. 目标用户
6. 术语表或风格约束
7. 技术约束，如占位符、长度限制、不可翻译片段
8. 可选：历史译法、相邻文案、截图说明

7.2 角色统一输出

每个角色的输出建议固定为：

1. 结论：通过 / 有保留通过 / 不通过
2. 主要问题：1 到 3 条
3. 修改建议：明确可执行
4. 推荐译文：如需修改则给出
5. 置信度：高 / 中 / 低

这样便于后续程序汇总和排序。

八、议会运行模式建议

8.1 轻量模式

适用场景
低风险、短文本、常规界面文案

建议参与角色
1. 忠实度审校员
2. 自然度润色员
3. 术语与一致性管理员
4. 技术与占位符审校员
5. 主编

8.2 标准模式

适用场景
大多数产品本地化任务

建议参与角色
全部角色或除品牌/风险角色外的核心角色，视项目情况启用

8.3 严格模式

适用场景
高曝光、风险较高或重要页面文案

建议参与角色
全部角色参与，并要求主编输出裁决说明与未采纳意见摘要

九、决策机制建议

1. 先独立评审，再汇总
避免角色互相影响，减少“跟风”意见。

2. 不以“票数”代替判断
多数意见可以作为信号，但不能覆盖明显的技术错误、术语冲突或风险问题。

3. 区分硬性问题与偏好问题
硬性问题包括变量损坏、语义错误、合规风险、术语冲突。
偏好问题包括语气更顺、表达更好看、品牌感更强。

4. 主编必须说明裁决理由
特别是在出现冲突时，应说明为何取舍。

5. 需要时输出多个候选
对于标题、按钮、营销文案等，可保留 A/B 候选并标明适用场景。

十、与仓库实现相关的落地建议

1. 角色定义应独立配置
每个角色应有清晰的名称、职责说明、评审重点和输出格式要求，便于后续扩展和替换。

2. 支持按内容类型选择角色集合
例如技术文档更强调忠实度、术语和技术安全；营销文案更强调品牌与自然度。

3. 输出必须结构化
即使最终展示为自然语言，也应保留结构字段，便于后续自动汇总、比对和回归测试。

4. 保留裁决痕迹
最终译文应能追溯到哪些角色提出了关键修改建议。

5. 支持术语表和风格指南注入
这是提高稳定性的关键，不应只靠通用模型发挥。

十一、验收标准

如果角色体系设计满足以下条件，可视为可进入实现阶段：

1. 每个角色职责边界清晰，无明显重复
2. 角色输出格式统一，便于程序消费
3. 存在默认优先级和冲突处理逻辑
4. 支持轻量、标准、严格三种运行模式
5. 能覆盖语义、术语、自然度、产品、技术、风险、品牌等主要维度
6. 能解释为什么最终采用某个译文
7. 能在仓库中被实现为配置化、可测试的议会机制

十二、建议的后续实现顺序

1. 先实现角色定义与统一输出格式
2. 再实现标准模式下的评审流程
3. 再加入主编汇总与裁决
4. 然后接入术语表、风格指南和上下文材料
5. 最后补充轻量模式、严格模式及回归评估

十三、总结

“本地化翻译议会”的核心不是让多个角色重复发表意见，而是把翻译质量拆解为可审查的多个维度，再通过统一输入、结构化输出和明确裁决规则，把分散判断变成稳定的翻译生产机制。

角色体系设计完成的标志，不是角色名字看起来完整，而是它们能在真实仓库里支撑一条可运行、可验证、可复用的本地化评审流程。
